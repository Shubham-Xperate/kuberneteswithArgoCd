# 14 — Source Control Workflow: Branching, PRs, and the GitHub + Azure DevOps Combination

## Where GitHub fits when Azure DevOps runs the pipeline

This project's pipeline lives in `.azuredevops/azure-pipelines.yml` and runs as an Azure DevOps Pipeline, but nothing about that requires the source code itself to be hosted in Azure Repos. **GitHub-for-source, Azure-Pipelines-for-CI/CD** is a genuinely common real-world combination, not a mismatched pairing — plenty of organizations standardize on GitHub for code review, issue tracking, and developer collaboration workflows while keeping Azure Pipelines (or another CI system entirely) as the thing that actually builds and ships the code, often because Azure Pipelines was already the established CI investment before GitHub became the preferred place for code, or because of specific integrations elsewhere in an Azure-heavy environment. The mechanism that connects the two is a **GitHub service connection** in Azure DevOps — conceptually identical to the ACR service connection covered in doc 13 (a credential stored once, centrally, referenced by name) except this one authorizes Azure Pipelines to read a GitHub repository, receive its webhook events (so a push or PR still triggers a pipeline run the same way it would from Azure Repos), and report build status back onto GitHub PRs. From the pipeline YAML's perspective, almost nothing changes — the same `trigger:`/`pr:` blocks, the same stages — only the `resources.repositories` section (or the pipeline's own linked-repository configuration in the Azure DevOps UI) points at GitHub instead of Azure Repos.

## Trunk-based development vs. GitFlow

Two dominant branching philosophies are worth contrasting before saying which one this project assumes. **GitFlow** organizes work around several long-lived branches with defined roles — typically `main` (or `master`) for released code, a separate `develop` branch where features integrate before a release, and dedicated `release/*` and `hotfix/*` branches for stabilizing a version before it ships. It was designed for a world of scheduled, versioned releases (think: ship a new version every few weeks, with a real code freeze/stabilization period beforehand) and gives you clear places to put in-progress work that isn't ready for release without touching the branch that represents "what's live." Its cost is complexity: more branches to keep in sync, more places for changes to get lost or duplicated, and merges between long-lived branches that can be genuinely painful the longer those branches diverge.

**Trunk-based development** — the model this project assumes — takes the opposite bet: there is exactly one long-lived branch, `main`, and every change is made on a short-lived feature branch that's merged back into `main` within, ideally, a day or two, via a pull request. There's no `develop` branch to keep in sync, no long-running feature branches accumulating weeks of drift from `main` before they're finally merged (and finally discover how much everything else has changed underneath them). The trade this makes is that it depends on `main` **always being deployable** — since there's no separate integration branch to catch problems before they reach the branch CI/CD treats as ground truth, the discipline of keeping `main` always in a shippable state has to be enforced some other way (small PRs, required reviews, a PR-validation pipeline that has to pass before merge — all covered below). This is precisely why trunk-based development pairs naturally with **continuous deployment to dev**: this project's pipeline (doc 13) auto-deploys every successful `main` build straight to the dev environment with no manual gate, which only makes sense as a policy if you can actually trust that `main` is safe to deploy at every commit — a trust that GitFlow's model, with its separate `develop` integration branch absorbing in-progress instability, doesn't require you to place directly on its equivalent of `main` in the same way.

## Branch protection rules: what they are and why each one exists

**Branch protection rules** (the same underlying concept exists on both GitHub and Azure Repos, under slightly different names) are settings enforced by the hosting platform itself — not by convention, not by a README asking people nicely — that restrict what can happen to a specific branch, typically `main`. Three rules matter most for this project's assumed workflow, and each solves a distinct failure mode:

**No direct pushes to `main`** — every change must arrive via a pull request, with direct `git push origin main` rejected by the platform regardless of who's pushing. This exists because a direct push bypasses every other safeguard (review, CI) entirely — it's not that people intend to push broken code directly, it's that removing the *possibility* is more reliable than trusting everyone to remember not to, especially under time pressure.

**Required reviews** — a PR cannot be merged until at least one (often more, for higher-risk repos) other person has approved it. This is the human half of quality control: a reviewer catches things automated checks structurally can't — is this the right approach, does this change interact badly with something the CI suite doesn't test, does this introduce a subtle security issue a linter wouldn't flag.

**Required PR-validation pipeline** — the merge button itself is disabled until the PR's associated pipeline run (in this project, the `Build_And_Test` stage triggered by the `pr:` block in `azure-pipelines.yml`, covered in doc 13) reports success. This is the automated half, catching the class of problems a human reviewer might miss on a visual read-through — a failing unit test, a build that doesn't compile, a linting violation — before the change ever reaches `main`.

Together, these three rules are the actual mechanism that makes "main is always deployable" something you can trust rather than something you merely hope is true — without them, trunk-based development's core assumption has no enforcement behind it at all.

## Monorepo vs. split repo, from the CI/CD angle

This project keeps `backend/`, `frontend/`, `helm/`, `terraform/`, and (as a lab simplification, per doc 08's discussion) `gitops/` all in one repository — a **monorepo** — rather than splitting them across several repos with independent lifecycles. From a CI/CD angle specifically, the tradeoff mirrors the split-repo-vs-monorepo GitOps discussion in `gitops/README.md`: a monorepo means one pull request can touch application code and infrastructure code together, reviewed and merged as a single, atomic unit, and a single pipeline run has visibility into everything at once — genuinely convenient for a project at this scale, and for the purposes of this lab, valuable for keeping everything in one place to learn from. The cost that shows up as repos and teams grow is that changes unrelated to a given area still show up in that area's commit history and can trigger broader pipeline runs than necessary (which is exactly why this pipeline's `paths.exclude` on `trigger:`/`pr:`, covered in doc 13, exists — a monorepo-specific concern a split-repo setup wouldn't need to think about at all), and it becomes harder to give different teams meaningfully different access controls over different parts of the codebase. Neither approach is objectively correct; it's the same kind of team-size/compliance/tooling-maturity tradeoff `gitops/README.md` describes for the GitOps repo split specifically, just applied one level up to the whole codebase.

## Commit and PR hygiene, briefly

A few small habits carry outsized value relative to their cost: commit messages that explain *why* a change was made, not just *what* changed (the diff already shows what changed); PRs kept small enough that a reviewer can actually hold the whole change in their head at once, which is also what makes short-lived branches practical in the first place; and a PR description that gives a reviewer the context they'd otherwise have to reconstruct by reading every changed file cold.

## Walking a real feature end to end

Bringing every piece from docs 08 and 13 together into one concrete story: a developer creates a short-lived branch off `main` — say `feature/add-order-history-endpoint` — and makes a series of commits implementing the change. They open a pull request targeting `main`. That PR being open automatically triggers the `pr:` block in `.azuredevops/azure-pipelines.yml`, running the `Build_And_Test` stage only — both the `.NET` and Angular jobs, in parallel, giving the developer fast feedback on whether their change actually builds and passes existing tests, with no risk of an image being pushed or a GitOps commit being made from code nobody's reviewed yet. Assuming that pipeline run succeeds, a teammate reviews the PR's diff, leaves comments or approves it, and — once both the required review and the required pipeline check are satisfied — the PR is merged into `main`.

That merge is a push to `main`, which triggers the full `trigger:` pipeline: `Build_And_Test` runs again (now against the merged result, not just the PR branch in isolation), then `Build_And_Push_Images` builds and pushes both Docker images to ACR tagged with the new `$(Build.BuildId)`, then `Update_GitOps_Dev` bumps `gitops/apps/ecommerce-dev/values.yaml` and commits. ArgoCD, watching that path (doc 08), notices the change on its next reconciliation pass and syncs the `ecommerce-dev` Application automatically — the new order-history endpoint is now running in the dev environment, with no human having done anything beyond writing code, reviewing a PR, and clicking merge. In parallel, `Update_GitOps_Prod` starts and immediately pauses at its `environment: production` approval gate (doc 13); some time later — after whatever validation in dev satisfies whoever's responsible for that decision — an authorized approver clicks approve, the stage bumps `gitops/apps/ecommerce-prod/values.yaml` and commits, and finally a human explicitly runs `argocd app sync ecommerce-prod` (or clicks Sync in the ArgoCD UI) to actually roll the change out to production — the second, deliberate manual step doc 08 describes as prod's second gate, distinct from and in addition to the pipeline's own approval.

## Key terms

- **Trunk-based development** — a branching model with one long-lived branch (`main`), short-lived feature branches merged back frequently via PRs, and an expectation that `main` is always deployable.
- **GitFlow** — a branching model with several long-lived branches (`main`, `develop`, `release/*`, `hotfix/*`) suited to scheduled, versioned releases.
- **Branch protection rule** — a platform-enforced restriction on a branch (e.g. no direct pushes, required reviews, required passing checks) that cannot be bypassed by convention alone.
- **Required review** — a branch protection rule requiring at least one other person's approval before a PR can merge.
- **PR-validation pipeline** — an automated build/test run triggered by a pull request, whose success is required before merging is allowed.
- **Service connection (GitHub)** — a stored credential in Azure DevOps authorizing it to read a GitHub repo, receive its events, and report status back to it, analogous to the ACR service connection in doc 13.
- **Monorepo vs. split repo** — keeping all related code (or, in doc 08's context, app code vs. GitOps config) in one repository versus splitting it across several with independent lifecycles and access controls.
